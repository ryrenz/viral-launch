# Viral Launch Scorecards

评分不是为了形式化，而是为了防止“听起来还行”的泛文案混进最终稿。

## Bold Claim Scorecard

每项 0-5 分。总分低于 18 不进入候选前三。任何 `Proof Risk = 5` 且无法补证据的 claim 直接淘汰。

| Dimension | Question | 0 | 5 |
|-----------|----------|---|---|
| Novelty | 是否让人觉得世界上出现了新东西？ | 像普通品类词 | 明显有新行为或新组合 |
| Pain Strength | 是否打中强痛点？ | 可有可无 | 用户已经在抱怨或付钱解决 |
| Specificity | 是否具体到动作、对象、结果？ | 抽象价值词 | 能看见用户做了什么 |
| Contrast | 是否能反衬旧世界很蠢或很慢？ | 没有对比 | 新旧行为差异明显 |
| Proof | 是否能被产品能力证明？ | 只能靠吹 | 有 demo、数据、截图或客户案例 |
| Timing | 是否解释为什么现在成立？ | 随时都能说 | 被近期技术/市场变化支撑 |
| Shareability | 是否容易被人转述？ | 复杂难复述 | 一句话能复述 |

额外标记：

- `Proof Risk`: 0-5，越高越危险。
- `Cliche Risk`: 0-5，越高越像竞品。

## Hook Scorecard

Hook 必须在第一行回答三件事：

- What is being launched?
- Why does it matter?
- Why now / why new?

淘汰标准：

- 以 `Excited to announce`、`Introducing` 开头但没有新信息。
- 任何公司换名字也能用。
- 只讲融资、团队、平台，不讲用户的新能力。
- 第一行没有冲突、好奇心、反差或具体结果。

保留标准：

- 读者第一眼知道这是什么。
- 读者知道为什么以前做不到或很难。
- 有具体对象、动作或结果。
- 不是靠夸张词撑起来。

## Body Scorecard

| Dimension | Check |
|-----------|-------|
| Old World | 是否展示旧行为的麻烦？ |
| New Behavior | 是否展示产品带来的新动作？ |
| Demo Clarity | 是否知道屏幕上应该出现什么？ |
| Proof | 是否有证据支撑 claim？ |
| Momentum | 每段是否推动欲望或理解？ |
| No Filler | 是否删除泛化解释？ |
| Category Shift | 是否让产品显得改变品类规则？ |

Body 的顺序优先：

1. Hook claim。
2. Old behavior。
3. Product action。
4. New behavior。
5. Proof。
6. CTA。

## Weapons Check

逐句检查。每行必须至少承担一个功能：

- 增加 novelty。
- 增加 proof。
- 增加 desire。
- 增加 clarity。
- 推动 action。

删除：

- “powerful”、“seamless”、“streamline”、“AI-powered”、“modern teams” 这类没有动作的词。
- 解释 demo 画面已经能展示的东西。
- 重复上一句信息。
- 任何公司都能说的话。
- 只表达愿景但没有证据的句子。

保留：

- 具体旧流程。
- 具体新动作。
- 用户原话。
- 数据或证据。
- 让品类重新被理解的句子。

## Controversy Check

目的不是制造争议，而是找到真实 tension。

可用 tension：

- 市场嘴上说 A，实际被 B 卡住。
- 用户以为需要更多工具，其实需要少一步。
- 旧品类承诺了 X，却一直交付 Y。
- 现有方案让用户接受了不合理成本。

不能做：

- 造假。
- 无证据攻击竞品。
- 为了“犀利”伤害目标用户。
- 把轻微不便包装成灾难。

## Technical Check

任何 claim 都必须过技术事实检查。

检查：

- “first”、“only”、“autonomous”、“agentic”、“real-time”、“production-ready” 是否有证据。
- demo 中暗示的能力是否真实存在。
- benchmark、速度、成本、准确率是否有来源。
- 是否混淆 prototype、beta、waitlist、GA。
- 是否把人工流程包装成全自动。

如果技术事实不确定：

- 实时搜索或查官方资料。
- 降级表述。
- 移到 human edit notes。

## Flow Check

读者不应该靠猜理解 demo。

检查：

- 开头有没有先建立看点。
- old behavior 是否足够短。
- new behavior 是否紧跟产品动作。
- proof 是否放在 claim 之后，而不是拖到最后。
- CTA 是否自然接住注意力。

常见问题：

- 先解释 30 秒才展示产品。
- 一次展示太多功能。
- 没有 aha moment。
- 从产品架构开始，而不是从用户痛点开始。

## Mom Test

假设读者非技术、耐心很短。

标红：

- 抽象名词没有具体动作。
- acronym 未解释。
- 句子里同时有 3 个以上概念。
- 需要懂行业背景才知道为什么重要。
- 代词指向不清。

改写方向：

- 把“提升效率”改成“少做哪一步”。
- 把“自动化 workflow”改成“用户点击什么，系统替他完成什么”。
- 把“平台”改成用户看到的界面和动作。

## CTA Check

CTA 必须和 launch goal 对齐。

| Goal | CTA |
|------|-----|
| Waitlist | Join / reply keyword / get early access |
| Demo calls | Book a demo / comment with use case |
| Product Hunt | Support us on Product Hunt |
| Feedback | Reply with workflow / problem / product |
| Distribution | Comment for teardown / template / checklist |

检查：

- 是否只有一个主 CTA。
- 是否太重。
- 是否有轻量行动给还没准备购买的人。
- 是否能制造评论而不显得像 engagement bait。

## Final Review

最终稿必须通过：

- 第一行有明确停留理由。
- Bold claim 没被正文稀释。
- Body 证明了 hook。
- 每个具体事实有来源。
- 没有泛化 SaaS 词。
- CTA 对齐目标。
- 非技术用户能懂。
- 人类编辑只需要做最后 5%，不是重写战略。
